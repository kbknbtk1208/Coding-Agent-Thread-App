# テストコード作成ガイドライン

## 目的

このガイドラインは、このリポジトリでテストコードを追加する際の最低限の統一ルールを定義するためのものです。実装者ごとにテスト手法や配置がばらつくことを避け、継続的に保守しやすい形でテストを追加できる状態を維持します。

## 使用するテストフレームワーク

- Unit Test には `Vitest` を使用すること
- 新規の Unit Test は `node:test` や他のテストフレームワークを使わず、`Vitest` に統一すること
- テストの実行は `npm run test`、ウォッチ実行は `npm run test:watch` を使用すること

## Unit Test の定義場所

- Unit Test はテスト対象の実装ファイルと同じディレクトリに配置すること
- ファイル名は `*.test.ts` とし、実装ファイルに対応する名前を使うこと
- 例:
  - `shared/domain/intermediate-segments.ts`
  - `shared/domain/intermediate-segments.test.ts`

配置ルールの意図は以下の通りです。

- 実装とテストの対応関係を見つけやすくする
- リファクタリング時に関連ファイルを追いやすくする
- 将来的に `shared/` 以外へ Unit Test を広げる場合でも同じルールで増やせるようにする

## 開発の進め方

- テスト追加を伴う実装は `TDD` で進めること
- 基本サイクルは `Red` → `Green` → `Refactor` とすること

### Red

- 先に期待する振る舞いをテストとして記述する
- その時点ではテストが失敗していてよい

### Green

- テストを通すための最小限の実装を追加する
- この段階では過剰な一般化を行わない

### Refactor

- テストが通っている状態を維持したまま実装とテストを整理する
- 重複除去、命名改善、責務整理を必要に応じて行う

## 補足

- 今回の Vitest 導入は最小構成であり、現時点では主に Node 環境で動く Unit Test を対象とする
- `renderer/` 向けの `jsdom` や Testing Library を使うテストは、必要になった時点で別途ガイドラインを追加または更新する
